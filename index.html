Spring Framework 개념

Feb 17, 2018 in Spring

Step #1

개요

  • Framework 의 정의

    • 어플리케이션을 구현하고 관리하는 환경, 틀, 약속. ex) 레고
    • Spring 의 철학에 대해 이해 필요
    • 유연성

  • Spring Framework

    • 설정 정보(XML)에 의해 어플리케이션을 구성하고, 객체를 생성하고, 생성된 객체를 관리하는 주체.

      설정 : XML or 어노테이션

Spring Framework 의 개념

  • 스프링은 자바 엔터프라이즈 어플리케이션의 개발에 사용되는 어플리케이션 프레임워크.

  • 어플리케이션 프레임워크는 어플리케이션 개발을 빠르고 효율적으로 할 수 있도록 어플리케이션의 바탕이 되는 틀과 공통 프로그래밍 모델, 그리고 API 등을 제공해준다.

    • 어플리케이션의 바탕이 되는 기본 틀 → 스프링 컨테이너

      • 스프링은 스프링 컨테이너 또는 어플리케이션 컨텍스트라고 불리는 스프링 런타임 엔진을 제공한다.
      • 스프링 컨테이너는 설정 정보 (XML : 배치기술서) 를 참고해서 어플리케이션을 구성하는 오브젝트(객체) 를 생성하고 관리한다.
      • 스프링 컨테이너는 독립적으로 동작할 수도 있지만 보통 웹 모듈에서 공작하는 서비스나 서블릿으로 등록해서 사용한다.
      • 스프링을 사용하려면 먼저 스프링 컨테이너를 다루는 방법과 스프링 컨테이너가 어플리케이션 오브젝트를 이용할 수 있도록 설정 정보를 작성하는 방법을 알아야 한다.
        • 설정 정보(XML) 에 의해 어플리케이션을 구성하고 객체를 생성, 관리하는 주체.

      홈페이지 vs 웹어플리케이션

      • 홈페이지 : 정적인 페이지, 동일한 결과(화면) 제공
      • 웹어플리케이션 : 사용자의 요청/응답에 따라, 각각 다른 화면(결과) 제공

    • 공통 프로그래밍 모델 → IoC/DI, 서비스 추상화, AOP

      제어 역전 : B클래스의 인스턴스가 제대로 생성되지 않을 경우, A클래스는 생성되지 않는다.

      의존성 주입

      • 프레임워크라는 어플리케이션을 구성하는 오브젝트(객체)가 생성되고 동작하는 방식에 대한 틀을 제공해줄 뿐 아니라 어플리케이션 코드가 어떻게 작성되어야 하는지에 대한 기준도 제시한다.

      • 이런 틀을 보통 프로그래밍 모델이라고 한다.

      • 스프링은 세가지 핵심 프로그래밍 모델을 제공한다.

        1. IoC/DI

          IoC/DI 라고 불리는 오브젝트 생명주기와 의존관계에 대한 프로그래밍 모델이다.

          스프링은 유연하고 확장성이 뛰어난 코드를 만들 수 있게 도와주는 객체지향 설계 원칙과 디자인 패텅의 핵심 원리를 담고 있는 IoC/DI 를 프레임워크의 근간으로 삼고 있다.

          • 유연성을 보장하는 기술적인 기반, 방법

          스프링 프레임워크에서 동작하는 코드는 IoC/DI 방식을 따라 작성되어야 스프링이 제공하는 가치를 제대로 누릴 수 있다.

          스프링이 직접 제공하는 모든 기술과 API, 심지어 컨테이너도 IoC/DI 방식으로 작성되어 있다.

          스프링을 바르게 이해하고 효율적으로 사용하는데 기본이 되며 가장 중요한 기술이다.

          • 객체지향 설계 (Function vs Method 의미 확인, 설계 → 디자인)
          • 자료구조, 알고리즘 (문제 해결 공식, 디자인 패턴과 의미 유사)

          • IoC

            • 객체 관리의 주체.

            • Inversion of Control : 제어의 역전

            • 객체 생성과 관리를 직접 하는 것이 아니라 외부(클래스의 밖, 컨테이너 안)에서 만들어진 오브젝트(객체) 를 제공받는 입장이 되는 것.

            • Bean : 스프링이 관리하는 대상.

            • BeanFactory : 객체를 관리하는 주체. 일반적으로 이 기능을 확장한 Application Context 를 사용한다.

              BeanFactory : 공장, Application Context : 공단

            • Application Context : Bean Factory 의 확장 버전.

            • 설정 정보 : 어떤 객체들이 관리 대생인지 설정한 정보. XML, Annotation 등을 사용한다. 객체 의존 관계도 전달하게 된다.

            • IoC 컨테이너 : 객체를 등록하고, 관리하는 공간. (스프링 컨테이너)

          • DI
            • 객체 관리의 방법.
            • 자바에서 객체를 생성하고 관리하는 방법은 모두 has-a 관계(포함 관계, 상속에서 언급)를 통해 처리해 왔기 때문에 이들은 모두 의존 관계가 치밀하고 밀접한 상황일 수 밖에 없다. 이로 인해… 이러한 강한 결합 상태는 유연성을 떨어뜨리는 결과를 초래하게 된다. 이러한 결합 상태를 풀어주는 과정(방법) 이 바로 DI 이다.
            • Dependency Injection : 의존성 주입 (의존관계 주입)
            • 의존 관계 주입 : 의존 객체를… 이 객체를 사용하는 객체에 외부에서 주입하는 액션 (상태)
            • 의존 관계 : A 가 실행되기 위해서 B 를 필요로 하는 상태
            • 의존 객체 : A 입장에서의 의존객체는 B (→ 주입되는 객체)
              • 우선… A 객체에 B 객체의 주소를 넘겨줌으로써 B 객체를 사용하게 된다면… A 객체는 여전히 B 객체를 사용하는 데에는 문제가 없고 B 객체는 A 객체와의 결합 상태가 느슨해진다고 할 수 있다.
              • 이 때, 넘겨주는 방법은 생성자 또는 setter() 메소드를 이용하는 방법이 있다.
              • DI 를 위해 객체는 작은 객체부터 생성해서 큰 객체로 DI 하게 된다.
              • 이 때, 완성된 오브젝트(객체) 가 만들어질 때 까지 객체를 관리할 주체가 필요한 상황이 되는데 그 역할을 수행하는 것이 바로 컨테이너인 것이다.

        2. 서비스 추상화

          스프링을 사용하면 환경이나 서버, 특정 기술에 종속되지 않고 이식성이 뛰어나며, 유연한 어플리케이션을 만들 수 있게 되는데 이를 가능하게 해주는 것이 바로 서비스 추상화이다.

          구체적인 기술과 환경에 종속되지 않도록 유연한 추상 계층을 두는 방법이다.

        3. AOP (Aspect Oriented Programming)

          이 AOP 는 어플리케이션 코드에 산재해서 나타나는 부가적인 기능을 독립적인 상태로 모듈화하는 프로그래밍 모델이다.

          스프링은 AOP 를 이용해서 다양한 엔터프라이즈 서비스를 적용하고도 깔끔한 코드를 유지할 수 있게 해준다.

          • 관점(관심사) 지향 프로그래밍
          • Primary Concerns(주업무), Cross Cutting Concerns(보조업무)

    • 기술 API

      • 스프링은 엔터프라이즈 어플리케이션 개발의 다양한 영역에 바로 활용할 수 있는 방대한 양의 기술 API 를 제공한다. UI 작성은 물론이고, 웹 프리젠테이션 계층, 비즈니스 서비스 계층, 기반 서비스 계층, 도메인 계층, 데이터 액세스 계층 등에서 필요한 주요 기술을 스프링에서 일관된 방식으로 사용할 수 있도록 지원해주는 기능과 전략 클래스를 제공한다.
      • 스프링이 제공하는 API 와 지원 기술은 모두 스프링의 핵심 프로그래밍 모델에 따라 작성되었기 때문에 이것들을 가져다 쓰는 것만으로도 스프링 프로그래밍 모델을 코드에 자연스럽게 적용할 수 있다. 스프링의 모든 기술은 표준 자바 엔터프라이즈 플랫폼(Java EE) 에 기술을 두고 있다.
      • 표준 기술과 더블어 유명 오픈소스 기술과 주요 상용 기술에 대한 지원 기능도 다양하게 제공된다.

스프링 프레임워크(Spring Framework) 의 주요 기능과 특징

  • 경량 컨테이너

    • 자바 객체를 담고, 이들 자바 객체의 생성과 소멸에 해당하는 라이프사이클을 관리

  • DI (Dependency Injection)

    • 의존성(관계) 주입(삽입) 을 지원.
    • 설정 파일을 통해 객체간의 의존관계를 정의.

  • AOP (Aspect Oriented Programming)

    • 관심사(관점) 지향 프로그래밍을 지원.
    • 여러 모듈에 공통적으로 쓰이는 기능을 분리하여 각 모듈에 적용하는 것이 가능.

  • POJO (Plain Old Java Object) 지원

    • 보통의 자바(빈) 객체
    • 단, 상속/인터페이스의 개념이 포함되지 않음
    • 가장 기본적인 자바 객체를 지원하여 기존 작성 코드의 유지성을 향상.

  • 트랜잭션 처리

    • JDBC 나 JTA(Java Transaction Application) 를 사용하는 설정 파일을 통해 트랜잭션을 관리하여 동일한 코드를 여러 환경에서 사용하도록 하는 것이 가능.

  • 다양한 프레임워크와 조합

    • myBatis, Hibernate, Struts, JPA, JDO, JMS, Quartz 등 다른 프레임워크 뿐 아니라 사용자가 만든 프레임워크와의 연동성을 제공한다.

Spring Framework 개념

 Feb 17, 2018

Step #1

개요

  • Framework 의 정의

    • 어플리케이션을 구현하고 관리하는 환경, 틀, 약속. ex) 레고
    • Spring 의 철학에 대해 이해 필요
    • 유연성

  • Spring Framework

    • 설정 정보(XML)에 의해 어플리케이션을 구성하고, 객체를 생성하고, 생성된 객체를 관리하는 주체.

      설정 : XML or 어노테이션

Spring Framework 의 개념

  • 스프링은 자바 엔터프라이즈 어플리케이션의 개발에 사용되는 어플리케이션 프레임워크.

  • 어플리케이션 프레임워크는 어플리케이션 개발을 빠르고 효율적으로 할 수 있도록 어플리케이션의 바탕이 되는 틀과 공통 프로그래밍 모델, 그리고 API 등을 제공해준다.

    • 어플리케이션의 바탕이 되는 기본 틀 → 스프링 컨테이너

      • 스프링은 스프링 컨테이너 또는 어플리케이션 컨텍스트라고 불리는 스프링 런타임 엔진을 제공한다.
      • 스프링 컨테이너는 설정 정보 (XML : 배치기술서) 를 참고해서 어플리케이션을 구성하는 오브젝트(객체) 를 생성하고 관리한다.
      • 스프링 컨테이너는 독립적으로 동작할 수도 있지만 보통 웹 모듈에서 공작하는 서비스나 서블릿으로 등록해서 사용한다.
      • 스프링을 사용하려면 먼저 스프링 컨테이너를 다루는 방법과 스프링 컨테이너가 어플리케이션 오브젝트를 이용할 수 있도록 설정 정보를 작성하는 방법을 알아야 한다.
        • 설정 정보(XML) 에 의해 어플리케이션을 구성하고 객체를 생성, 관리하는 주체.

      홈페이지 vs 웹어플리케이션

      • 홈페이지 : 정적인 페이지, 동일한 결과(화면) 제공
      • 웹어플리케이션 : 사용자의 요청/응답에 따라, 각각 다른 화면(결과) 제공

    • 공통 프로그래밍 모델 → IoC/DI, 서비스 추상화, AOP

      제어 역전 : B클래스의 인스턴스가 제대로 생성되지 않을 경우, A클래스는 생성되지 않는다.

      의존성 주입

      • 프레임워크라는 어플리케이션을 구성하는 오브젝트(객체)가 생성되고 동작하는 방식에 대한 틀을 제공해줄 뿐 아니라 어플리케이션 코드가 어떻게 작성되어야 하는지에 대한 기준도 제시한다.

      • 이런 틀을 보통 프로그래밍 모델이라고 한다.

      • 스프링은 세가지 핵심 프로그래밍 모델을 제공한다.

        1. IoC/DI

          IoC/DI 라고 불리는 오브젝트 생명주기와 의존관계에 대한 프로그래밍 모델이다.

          스프링은 유연하고 확장성이 뛰어난 코드를 만들 수 있게 도와주는 객체지향 설계 원칙과 디자인 패텅의 핵심 원리를 담고 있는 IoC/DI 를 프레임워크의 근간으로 삼고 있다.

          • 유연성을 보장하는 기술적인 기반, 방법

          스프링 프레임워크에서 동작하는 코드는 IoC/DI 방식을 따라 작성되어야 스프링이 제공하는 가치를 제대로 누릴 수 있다.

          스프링이 직접 제공하는 모든 기술과 API, 심지어 컨테이너도 IoC/DI 방식으로 작성되어 있다.

          스프링을 바르게 이해하고 효율적으로 사용하는데 기본이 되며 가장 중요한 기술이다.

          • 객체지향 설계 (Function vs Method 의미 확인, 설계 → 디자인)
          • 자료구조, 알고리즘 (문제 해결 공식, 디자인 패턴과 의미 유사)

          • IoC

            • 객체 관리의 주체.

            • Inversion of Control : 제어의 역전

            • 객체 생성과 관리를 직접 하는 것이 아니라 외부(클래스의 밖, 컨테이너 안)에서 만들어진 오브젝트(객체) 를 제공받는 입장이 되는 것.

            • Bean : 스프링이 관리하는 대상.

            • BeanFactory : 객체를 관리하는 주체. 일반적으로 이 기능을 확장한 Application Context 를 사용한다.

              BeanFactory : 공장, Application Context : 공단

            • Application Context : Bean Factory 의 확장 버전.

            • 설정 정보 : 어떤 객체들이 관리 대생인지 설정한 정보. XML, Annotation 등을 사용한다. 객체 의존 관계도 전달하게 된다.

            • IoC 컨테이너 : 객체를 등록하고, 관리하는 공간. (스프링 컨테이너)

          • DI
            • 객체 관리의 방법.
            • 자바에서 객체를 생성하고 관리하는 방법은 모두 has-a 관계(포함 관계, 상속에서 언급)를 통해 처리해 왔기 때문에 이들은 모두 의존 관계가 치밀하고 밀접한 상황일 수 밖에 없다. 이로 인해… 이러한 강한 결합 상태는 유연성을 떨어뜨리는 결과를 초래하게 된다. 이러한 결합 상태를 풀어주는 과정(방법) 이 바로 DI 이다.
            • Dependency Injection : 의존성 주입 (의존관계 주입)
            • 의존 관계 주입 : 의존 객체를… 이 객체를 사용하는 객체에 외부에서 주입하는 액션 (상태)
            • 의존 관계 : A 가 실행되기 위해서 B 를 필요로 하는 상태
            • 의존 객체 : A 입장에서의 의존객체는 B (→ 주입되는 객체)
              • 우선… A 객체에 B 객체의 주소를 넘겨줌으로써 B 객체를 사용하게 된다면… A 객체는 여전히 B 객체를 사용하는 데에는 문제가 없고 B 객체는 A 객체와의 결합 상태가 느슨해진다고 할 수 있다.
              • 이 때, 넘겨주는 방법은 생성자 또는 setter() 메소드를 이용하는 방법이 있다.
              • DI 를 위해 객체는 작은 객체부터 생성해서 큰 객체로 DI 하게 된다.
              • 이 때, 완성된 오브젝트(객체) 가 만들어질 때 까지 객체를 관리할 주체가 필요한 상황이 되는데 그 역할을 수행하는 것이 바로 컨테이너인 것이다.

        2. 서비스 추상화

          스프링을 사용하면 환경이나 서버, 특정 기술에 종속되지 않고 이식성이 뛰어나며, 유연한 어플리케이션을 만들 수 있게 되는데 이를 가능하게 해주는 것이 바로 서비스 추상화이다.

          구체적인 기술과 환경에 종속되지 않도록 유연한 추상 계층을 두는 방법이다.

        3. AOP (Aspect Oriented Programming)

          이 AOP 는 어플리케이션 코드에 산재해서 나타나는 부가적인 기능을 독립적인 상태로 모듈화하는 프로그래밍 모델이다.

          스프링은 AOP 를 이용해서 다양한 엔터프라이즈 서비스를 적용하고도 깔끔한 코드를 유지할 수 있게 해준다.

          • 관점(관심사) 지향 프로그래밍
          • Primary Concerns(주업무), Cross Cutting Concerns(보조업무)

    • 기술 API

      • 스프링은 엔터프라이즈 어플리케이션 개발의 다양한 영역에 바로 활용할 수 있는 방대한 양의 기술 API 를 제공한다. UI 작성은 물론이고, 웹 프리젠테이션 계층, 비즈니스 서비스 계층, 기반 서비스 계층, 도메인 계층, 데이터 액세스 계층 등에서 필요한 주요 기술을 스프링에서 일관된 방식으로 사용할 수 있도록 지원해주는 기능과 전략 클래스를 제공한다.
      • 스프링이 제공하는 API 와 지원 기술은 모두 스프링의 핵심 프로그래밍 모델에 따라 작성되었기 때문에 이것들을 가져다 쓰는 것만으로도 스프링 프로그래밍 모델을 코드에 자연스럽게 적용할 수 있다. 스프링의 모든 기술은 표준 자바 엔터프라이즈 플랫폼(Java EE) 에 기술을 두고 있다.
      • 표준 기술과 더블어 유명 오픈소스 기술과 주요 상용 기술에 대한 지원 기능도 다양하게 제공된다.

스프링 프레임워크(Spring Framework) 의 주요 기능과 특징

  • 경량 컨테이너

    • 자바 객체를 담고, 이들 자바 객체의 생성과 소멸에 해당하는 라이프사이클을 관리

  • DI (Dependency Injection)

    • 의존성(관계) 주입(삽입) 을 지원.
    • 설정 파일을 통해 객체간의 의존관계를 정의.

  • AOP (Aspect Oriented Programming)

    • 관심사(관점) 지향 프로그래밍을 지원.
    • 여러 모듈에 공통적으로 쓰이는 기능을 분리하여 각 모듈에 적용하는 것이 가능.

  • POJO (Plain Old Java Object) 지원

    • 보통의 자바(빈) 객체
    • 단, 상속/인터페이스의 개념이 포함되지 않음
    • 가장 기본적인 자바 객체를 지원하여 기존 작성 코드의 유지성을 향상.

  • 트랜잭션 처리

    • JDBC 나 JTA(Java Transaction Application) 를 사용하는 설정 파일을 통해 트랜잭션을 관리하여 동일한 코드를 여러 환경에서 사용하도록 하는 것이 가능.

  • 다양한 프레임워크와 조합

    • myBatis, Hibernate, Struts, JPA, JDO, JMS, Quartz 등 다른 프레임워크 뿐 아니라 사용자가 만든 프레임워크와의 연동성을 제공한다.

리눅스 Shell 프로그래밍

    • 새로 만들기 > 메모리, 2048Mu > 기존 가상 하드 디스크 사용, ubuntu1604

  • 공유 폴더 추가

    • 설정 > 공유 폴더 > 추가, share (자동 마운트)

  • 계정 : linux / ubuntu

  • Linux System 주요 구성 요소

    • computer hardware > Linux kernel > Middle (Library, Apis) > Application
    • 심도 깊은 Application 을 개발하기 위해서는, kernel 에 대한 이해도 있어야 함

  • 터미널 폰트 사이즈 조정 : ctrl + shift + (+ / -)

  • 로그인 계정의 home 디렉토리

    • ~ 로 표시됨

  • 디렉토리 생성

    • 모든 디렉토리에는 . 과 .. 디렉토리가 숨겨져 있음

  • 홈 디렉토리에 숨겨져 있는 shell program

    • .bashrc > Command 를 해석하고 실행하는 소프트웨어, Command 창이 생성될 때 실행
    • .bash_logout > Command 창이 닫힐 떼 실행
    • .profile > 로그인 할때

  • Run Level

    • 리눅스의 부팅 모드 > runlevel, N 5 (previous mode / current mode)

  • 리눅스의 Default Shell > bash

    • 유닉스의 sh, csh, ksh, zsh 의 장점을 가지고 개발
    • 기본적인 Command 는 비슷하나, 심도 깊은 Command 는 달라짐 (Admin 등등에서 사용하는)

  • Shell 확인

    • ps

    • Command 가 새로 띄워질 때마다 bash 이 새로 실행됨, 따라서 실행 중인 bash 의 PID 가 다름

  • Command 창 띄우기

    • 탭 열기 : ctrl + shift + t
    • 새 창 열기 : alt + shift + t

  • 사용자 추가 시, default shell 지정

  • bin(기본 실행파일), sbin(super 유저), usr/bin(설치)

  • which > 파일 위치 확인

  • dash 실행

  • shell 프로그램 실행 parent 확인

  • man > 동일한 Command 일 경우, Section 번호를 지정한다.

  • grep

  • touch

  • 리눅스 파일시스템 계층 표준 (FHS : Filesystem Hierarchy Standard)

    • 추가 공부 필요

  • 매뉴얼에 대한 설명

    • man7.org > https://www.kernel.org/doc/man-pages (The Linux man-pages project)

  • ps 계층 구조

  • sleep > foreground 실행 > Command 실행, delay 주기

  • alias

    • 실행된 Command 창에서만 alias 가 적용됨
    • 모든 Command 창에서, alias 가 적용되게 하려면, .bashrc 를 수정한다

Step2

  • ~ (home directory)

    • 계정의 home directory 를 알고 싶을 때

  • alias

  • gedit, .bashrc 수정, 반영 (source)

    • .bashrc 를 수정하였을 경우, source 를 하거나, 터미널을 다시 열면 반영됨
    • 새로운 터미널을 열 경우, source 가 실행됨. .bashrc 쉘 프로그램이 실행됨

  • .profile

    • alias 를 추가할 수 있음
    • .profile 은 로그인 하는 시점에 반영 됨 (.bashrc 는 새 터미널을 열었을 때 반영)
    • 반영 시점이 다르기 때문에, 적절한 시점에 반영 하도록 프로그래밍 하면 됨

  • history

  • .bashrc > rc 의 의미는 Run Command, 실행 Command

  • /etc/profile 에 .bashrc 또는 .profile 의 template 파일이 존재 함

    • home 디렉토리의 파일을 작업하다가, 잘 못 되었을 경우, 해당 파일을 이용하여 복구 함
    • 배포판에 따라 경로가 다를 수 있음

  • bash 에 설정되어 있는 환경 변수 확인

    • SHELL, USER, PWD, HOME, PATH 등 환경 변수를 많이 사용함
    • PATH 라는 환경 변수는, 명령어 실행 시, 명령이 저장된 경로를 생략할 수 있도록 함

  • 파일 이동

  • echo

  • permission

    • Default 권한은, 파일을 생성하면, 실행 권한이 없음
    • Default 권한 변경은 umask 로 변경함 > .bashrc 에 추가 함
      • http://brothernsister.tistory.com/35

  • 사용자가 작성한 shell 프로그램은 sub shell 에서 동작함

    • 로그인 shell 이 아니라, 새로은 bash 이 동작하면서 실행 함
    • 아주 중요한 개념

  • sh 01.sh 로 실행 시, 주의 사항

    • sh 가 어떤 shell 로 링크되어 있는지 확인해야 함
    • ubuntu 는 sh 이 dash 로 링크되어 있음, 따라서 bash 문법의 shell 프로그램은 sh 로 실행 시 오류가 발생할 수 있음
    • bash 01.sh 로 실행하는 것과 동일한 의미

  • Sub Shell 실습

    • Command 에 () 를 추가하면, Sub Shell 로 실행됨
    • () 내에 () 추가 가능

  • 환경 변수 수정

  • 사용자가 작성한 shell 프로그램이 여러개일 경우, home 디렉토리에 bin 디렉토리를 추가하고 통합 관리하는 것이 편함

  • 환경 변수 추가, Sub Shell 에서도 추가한 환경 변수가 반영되도록 하려면

  • . 이나 source 로 shell 을 반영하는 것은 로그인 shell 로 실행하는 것

    • bash .bashrc 와 같이 Sub shell 로 실행하게 되면, 환경 변수가 반영되지 않음
    • 로그인 shell 로 실행해야 될 것과, Sub shell 로 실행해야 될 것을 구분해야 함
    • 내 환경을 전체적으로 변경할 때, sourch 로 반영, 환경에 영향을 주고 싶지 않을 때, Sub shell 로 반영

  • 환경 변수 삭제, unset

  • prompt

  • id, UID

  • IFS (Inter Field Seperator) > argument 의 구분자, space

  • $? > shell Command 의 실행 결과, 종료 상태

  • shell 프로그램 작성 (환경 변수 추가), 04.user_var.sh

  • shell 프로그램 작성 (산술연산)

  • bc 는 calculator Application 이다.

    • 종료할 때는, ctrl + d 또는 quit 를 사용함

  • shell 프로그램 작성 (argument 값 전달)

  • init 0 > power off
